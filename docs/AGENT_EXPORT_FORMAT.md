# Agent Export Format Specification
**Version:** 1.0
**Date:** 2026-03-12
**Status:** Approved for Sprint 4 implementation

This document defines the canonical format for exporting and importing agent profiles in Clowder v2. Two formats are supported: JSON (machine-readable, full fidelity) and Markdown (human-readable summary). Both are generated by `GET /api/agents/{name}/export?format=json|md` and accepted by `POST /api/agents/import`.

---

## JSON Export Schema

```json
{
  "name": "string — agent identifier (primary key, alphanumeric + dash + underscore, max 64 chars)",
  "role": "string — agent role (e.g. coder, reviewer, security)",
  "model": "string — LLM model name used by this agent",
  "bio": "string | null — free-form agent biography",
  "skills": ["string"],
  "personality_seed": "string | null — system prompt prefix that seeds personality",
  "rag_config": {
    "source_types": ["string"],
    "team_scope": "string | null",
    "top_k": "integer"
  },
  "preferences": {
    "verbose": "boolean",
    "context_window_preference": "string | null"
  },
  "system_prompt": "string — full system prompt as assembled at export time (computed from personality_seed + rag_config; not a stored column)",
  "avatar_svg": "string | null — SVG text (max 50 KB); null if no custom avatar",
  "exported_at": "ISO 8601 timestamp — e.g. 2026-03-12T14:30:00Z",
  "exported_from": "string — board identifier or hostname (informational)",
  "context_history": [
    {
      "source_type": "string — e.g. task_result, chat_message",
      "content": "string — snippet of past context (max 500 chars)",
      "created_at": "ISO 8601 timestamp"
    }
  ]
}
```

### Field Notes

- **`system_prompt`** is a computed field assembled at export time. It is not stored as a separate database column. The stored source of truth is `personality_seed`. On import, if `personality_seed` is present, it is stored directly; `system_prompt` is ignored (it can be reconstructed).
- **`rag_config`** controls what the agent auto-retrieves when claiming tasks. If absent from the model at export time (column not yet added), the field is exported as `{}`.
- **`context_history`** is a derived field — queried from `context_items` filtered by `agent_id` at export time. It is informational and is not imported (the target board has its own context store). Maximum 20 entries exported, ordered by `created_at` descending.
- **`avatar_svg`** is included inline as SVG text. Importers must validate size (≤ 50 KB) and XML well-formedness before storing.

## Markdown Export Format

### Section Structure

```markdown
# {name}

**Role:** {role}
**Model:** {model}
**Bio:** {bio | "(none)"}

## Skills

- {skill 1}
- {skill 2}
- ...

## Personality Seed

{personality_seed | "(none)"}

## RAG Configuration

- Source types: {rag_config.source_types | "default"}
- Team scope: {rag_config.team_scope | "global"}
- Top-K context items: {rag_config.top_k | 5}

---
*Exported from Clowder v2 at {exported_at}*
```

### Markdown Rules

- Agent `name` is always an H1 heading (`#`).
- `Role`, `Model`, and `Bio` are bold labels followed by their value on the same line.
- `Skills` is an H2 section (`##`) containing a bullet list. If skills is empty or null, write `- (none)`.
- `Personality Seed` is an H2 section containing the raw seed text as a paragraph. If null, write `(none)`.
- `RAG Configuration` is an H2 section with a bullet list of key settings.
- A horizontal rule (`---`) and an italicised export timestamp close the document.
- Avatar SVG is **not included** in the Markdown export (binary/SVG data is JSON-only).
- Context history is **not included** in the Markdown export.

## Import Rules

### Required Fields

The following fields are required for a valid import. An import request missing any of them returns HTTP 422 with a field-level error message.

| Field | Reason required |
|-------|----------------|
| `name` | Primary key; uniqueness check and upsert target |
| `role` | Non-nullable in the `agents` schema |

All other fields are optional and default to their model defaults if absent (`model` → `"unknown"`, `bio` → `null`, `skills` → `null`, etc.).

### Validation Rules

**`name` validation:**
- Must match regex `^[a-zA-Z0-9_-]{1,64}$`
- Maximum 64 characters
- If the name fails validation: HTTP 422 `{"detail": "name must be alphanumeric, dash, or underscore (max 64 chars)"}`

**`avatar_svg` validation:**
- Must be valid XML (parsed with `xml.etree.ElementTree`; rejected if malformed)
- Encoded UTF-8 size must be ≤ 50,000 bytes (50 KB)
- If invalid XML: HTTP 422 `{"detail": "avatar_svg is not valid SVG XML"}`
- If oversized: HTTP 422 `{"detail": "avatar_svg exceeds 50 KB limit"}`

**`skills` validation:**
- Must be a JSON array of strings if present
- Individual skill strings: max 64 chars each, no empty strings
- Maximum 50 skills per agent

**`personality_seed` validation:**
- Text, max 8,000 characters
- If `system_prompt` is also present in the payload, `personality_seed` takes precedence; `system_prompt` is discarded

**`rag_config` validation:**
- Must be a JSON object if present
- Unknown keys are silently ignored (forward-compatible)
- `top_k`: integer 1–20 if specified

### Name Collision Behaviour (Upsert Policy)

When an agent with the given `name` already exists in the database:

- The import **updates** the existing record (upsert). It does not error.
- Fields present in the import payload overwrite the stored value.
- Fields absent from the import payload are **left unchanged** (partial update semantics).
- HTTP response: `{"ok": true, "created": false, "name": "{name}"}` (HTTP 200)

When no agent with the given `name` exists:

- A new agent record is created.
- HTTP response: `{"ok": true, "created": true, "name": "{name}"}` (HTTP 201)

### Immutable Fields (Active Agents)

The following fields are **not overwritten** by import when the agent's `status` is `"online"` (i.e., the agent is currently running):

| Field | Reason |
|-------|--------|
| `model` | Changing the model under a running agent would produce inconsistent token accounting and behaviour mid-task |
| `status` | Status is controlled by heartbeat; import cannot force an online agent offline |
| `team` | Team assignment is managed by the Team Leader, not the import flow |

If the agent is `"offline"` or `"fired"`, all fields including `model` are writable on import.

### Fields Ignored on Import

| Field | Reason |
|-------|--------|
| `system_prompt` | Computed at export time; `personality_seed` is the stored field |
| `context_history` | Target board has its own context store; imported history is not re-indexed |
| `exported_at` | Export metadata; not stored |
| `exported_from` | Export metadata; not stored |

## API Endpoints (for Claude to implement)

### Export

```
GET /api/agents/{name}/export?format=json
GET /api/agents/{name}/export?format=md
```

- Returns the agent export in the requested format.
- `format` defaults to `json` if omitted.
- For `md`: Content-Type `text/markdown; charset=utf-8`.
- For `json`: Content-Type `application/json`.
- Also persists the rendered export artifact to `agent_exports` table (model: `AgentExport`).
- HTTP 404 if agent not found.

### Import

```
POST /api/agents/import
Content-Type: application/json
```

- Request body: the JSON export schema (or a subset; only `name` and `role` required).
- Response: `{"ok": true, "created": bool, "name": str}` HTTP 200/201.
- Response on validation failure: HTTP 422 with Pydantic-style field errors.

## Implementation Notes for `backend/api/exports.py`

- Use Pydantic v2 `model_validator` for cross-field validation (e.g. `system_prompt` discard when `personality_seed` present).
- Avatar SVG validation must use `xml.etree.ElementTree.fromstring()` inside a `try/except ET.ParseError`.
- Size check must be `len(avatar_svg.encode("utf-8")) <= 50_000` (not `len(avatar_svg)`).
- The upsert on import should use SQLAlchemy's `merge()` (or explicit `db.get()` + field assignment) rather than a raw INSERT ON CONFLICT, to respect the immutable-field rules for online agents.
- Context history export: join `context_items` where `source_id` matches any `retrieval_logs.agent_id == name`, or tag context items with the claiming agent. Confirm the exact FK relationship with Qwen before implementing.
